## TreeTalk 是什么

TreeTalk 是一个面向学习、资料理解、代码分析和复杂问题探索的 AI 桌面客户端。

普通 AI 聊天工具通常把所有问题和回答放在一条不断变长的时间线上。随着追问次数增加，用户很容易遇到这些问题：

- 想深入研究回答中的一个知识点，却会打断原来的问题；
- 追问很多层以后，很难返回最初的问题；
- 多个研究方向混在同一个聊天记录中；
- 想针对一行代码、一个公式或一句话提问时，需要反复复制上下文；
- 聊天结束后只剩一条很长的记录，难以看出自己的思考路径；
- 有价值的问答无法方便地保存为长期知识材料。

TreeTalk 使用**问题树、层级追问和本地知识仓库**重新组织这个过程。

```text
传统聊天：问题 → 回答 → 问题 → 回答 → 问题 → 回答……

TreeTalk：
主问题
├── 子问题 A
│   ├── 深入问题 A-1
│   └── 深入问题 A-2
├── 子问题 B
└── 子问题 C
```

你可以从任意回答中选择一个知识点继续追问，完成后返回父问题，再进入另一个分支。最终保存的不只是一个答案，而是完整的知识探索路径。

## 产品定位

TreeTalk 的目标不是替代 Obsidian、Notion、Logseq 等专业笔记软件。

它更关注 AI 获取知识的前半段：

1. 提出问题；
2. 阅读回答；
3. 发现不理解或值得继续研究的内容；
4. 针对局部内容继续追问；
5. 在不同问题分支之间切换；
6. 将有价值的问题结构保存下来；
7. 再根据需要整理到其他笔记软件中。

因此，TreeTalk 更像是一个**结构化 AI 对话工作台**，而不是传统意义上的笔记编辑器。

## TreeTalk 解决了什么需求

### 1. 解决 AI 对话只能线性发展的限制

一个回答往往包含多个知识点。例如，学习链表时，AI 可能同时讲到节点、指针、内存和遍历。

在普通聊天工具中，继续追问“指针”以后，再研究“遍历”就容易与之前的内容混在一起。TreeTalk 会为不同方向建立独立的问题分支，让每个方向都有自己的上下文。

### 2. 解决深入追问后无法快速返回的问题

TreeTalk 会记录当前问题与父问题之间的关系。

用户可以：

- 进入一个子问题；
- 连续深入多个层级；
- 完成当前分支后返回上一级；
- 通过问题树直接跳转到任意历史节点；
- 在原节点上继续建立新的分支。

### 3. 解决局部内容追问不准确的问题

很多提问并不是针对整篇回答，而是针对其中某一部分，例如：

- “这行代码为什么这样写？”
- “这个公式从哪里来的？”
- “这里的指针指向什么？”
- “这句话和上面的结论有什么关系？”

TreeTalk 支持框选文字、公式或代码后继续提问，并在提问时保留父问题、父回答、选中的原文和新问题，使 AI 能准确理解用户所说的“这里”是什么。

### 4. 解决长对话中知识结构不清晰的问题

传统聊天记录强调时间顺序，但学习更关注逻辑关系。

TreeTalk 通过问题树展示：

- 哪个问题是主问题；
- 哪些问题是从同一回答延伸出来的；
- 当前正在研究哪个分支；
- 每个结论是通过怎样的追问得到的。

### 5. 解决 AI 对话难以长期沉淀的问题

TreeTalk 可以将有价值的会话保存到本地知识仓库。

保存内容不仅包括最终回答，也包括：

- 主问题；
- 子问题；
- 问题之间的父子关系；
- 每个节点对应的回答；
- 与问题关联的附件；
- 完整的学习和推理路径。

### 6. 解决不同 AI 使用方式的选择问题

TreeTalk 支持两种 AI 接入思路：

- **API 模式**：适合拥有模型 API Key、希望自行选择服务和模型的用户；
- **浏览器桥接模式**：适合希望使用 ChatGPT 网页账户、不想单独配置 API 的用户。

## 核心使用流程

```text
提出一个主问题
      ↓
AI 生成回答
      ↓
阅读并框选一个知识点
      ↓
针对选中内容建立子问题
      ↓
继续深入或返回父问题
      ↓
进入其他问题分支
      ↓
将完整问题树保存到知识仓库
```

例如，在学习 C++ 链表时，可以形成下面的问题结构：

```text
链表是什么？
├── 节点中的 next 是什么？
│   ├── 指针变量保存的是什么？
│   └── nullptr 表示什么？
├── 怎么遍历链表？
│   └── p = p->next 为什么能移动？
└── 链表和数组有什么区别？
```

每个节点都保留自己的回答和上下文，用户不需要反复复制之前的聊天内容。

## 主要功能

### 问题树与层级追问

- 围绕主问题创建多个子问题；
- 自动记录父问题和子问题关系；
- 支持连续深入多层问题；
- 支持返回父问题；
- 支持从问题树直接进入任意节点；
- 避免多个研究方向混在同一条聊天记录中。

### 框选追问

- 支持框选普通文字；
- 支持框选代码片段；
- 支持针对数学公式继续提问；
- 自动携带相关上下文；
- 减少复制、粘贴和重新解释背景的操作。

### 逻辑树与问题图谱

- 可视化显示问题之间的逻辑关系；
- 查看完整问题分支；
- 快速识别当前问题的位置；
- 适合梳理课程知识、代码逻辑、数学推导和复杂概念。

### 流式回答

- 支持回答逐步生成；
- 显示生成状态；
- 用户可以在生成过程中阅读已有内容；
- 切换问题节点时，后台任务可以继续处理；
- 返回节点后仍可查看对应的生成结果。

### 代码与公式显示

- 支持 Markdown 内容；
- 支持代码块和行内代码；
- 使用 KaTeX 渲染数学公式；
- 适合编程、数学、数据结构、网络工程和理科学习场景。

### 图片与附件

- 支持从输入区添加附件；
- 支持直接粘贴图片；
- 可以将附件与当前问题一起发送；
- 可用于分析截图、课程资料、代码图片和常见文档；
- 附件内容可以继续作为问题树的一部分进行追问。

### 本地知识仓库

- 将工作台中的问题树保存为知识文件；
- 使用文件夹组织不同主题；
- 支持文件和文件夹重命名；
- 支持删除不需要的知识内容；
- 打开知识文件后可以继续追问；
- 临时工作台和长期知识仓库相互独立。

### 多问题并行

- 切换到其他问题时，当前回答可在后台继续生成；
- 不同问题分支不会互相覆盖；
- 适合同时分析多个相关知识点；
- 减少等待回答时无法进行其他操作的问题。

### API 模式

- 支持配置模型服务；
- 适合拥有 API Key 的用户；
- 可以根据需要切换模型或服务；
- 配置和会话数据保存在本地设备中。

### 浏览器桥接模式

- 打开独立的 ChatGPT 登录窗口；
- 用户可以使用自己的网页账户登录；
- TreeTalk 将问题和附件传递到网页输入区域；
- 等待网页回复完成后，将内容回传到对应问题节点；
- 登录窗口保留普通系统窗口控制，方便用户正常操作网页。

### 阅读与交互体验

- 问题节点切换使用平滑过渡；
- 阅读长回答时不会强制滚动到底部；
- 主窗口提供自定义窗口控制；
- 支持调整辅助区域的显示大小；
- 适合长时间阅读和连续研究。

## 适合哪些人

TreeTalk 比较适合：

- 使用 AI 学习编程、数学、数据结构和网络技术的学生；
- 需要阅读论文、课程资料或技术文档的用户；
- 分析代码、报错和复杂逻辑的开发者；
- 希望保留完整排查过程的技术人员；
- 想利用 AI 获取知识，但希望将正式笔记交给其他笔记软件的用户；
- 认为普通 AI 聊天记录太长、分支关系不清晰的重度用户。

TreeTalk 不以多人协作、在线文档编辑或完整笔记数据库为主要目标。它主要解决的是：**如何更有结构地向 AI 提问、理解回答和沉淀问题路径。**

## 与普通 AI 聊天界面的区别

| 普通 AI 聊天 | TreeTalk |
|---|---|
| 对话从上到下不断增长 | 问题按父子关系组织 |
| 深入一个方向后不方便返回 | 可以返回父问题或点击树节点 |
| 多个研究方向混在同一会话 | 每个方向形成独立分支 |
| 经常需要复制原文再追问 | 可以直接框选内容追问 |
| 主要保存聊天时间线 | 保存问题树和知识探索路径 |
| 长对话容易丢失重点 | 当前问题和所在路径始终明确 |
| 最终只留下大量聊天文本 | 可以沉淀为本地知识文件 |

## macOS 窗口行为

TreeTalk 在 macOS 上采用混合窗口设计：

- TreeTalk 主窗口使用自定义无边框窗口；
- 主窗口不显示左上角红、黄、绿三个系统按钮；
- 主窗口继续使用界面右上角的最小化、最大化和关闭按钮；
- ChatGPT 登录窗口保留 macOS 原生红、黄、绿按钮；
- 关闭主窗口后，应用可继续保留在 Dock，并能重新打开窗口。

当前 macOS 发布包可能没有经过 Apple Developer ID 签名。首次运行时，可能需要在 Finder 中右键应用并选择“打开”。

## 数据与隐私

TreeTalk 采用本地优先设计：

- 学习会话默认保存在用户设备中；
- 知识仓库默认保存在本地；
- 附件与问题树不会因为普通安装或升级被主动删除；
- API Key、登录凭据和个人知识内容不应提交到 GitHub Issues；
- 使用第三方模型 API 或 ChatGPT 网页时，发送的内容仍受对应服务隐私政策约束。

公开反馈问题前，请先移除日志或截图中的 API Key、账号信息、私人文档和敏感路径。
